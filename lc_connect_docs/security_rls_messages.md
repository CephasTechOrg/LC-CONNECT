# Messages Table — Row Level Security (RLS) Documentation

**Date implemented:** 2026-05-10  
**Implemented by:** Cephas Bonsu-Osei  
**Status:** Active and verified

---

## Why this exists

LC Connect uses Supabase Realtime so that new chat messages appear instantly inside an open conversation without the user needing to refresh. To make this work, the Flutter app initialises a Supabase client using the project's **anon key**, which is embedded inside the compiled app bundle.

The anon key is not a secret — anyone can extract it from an Android APK using tools like `apktool`, or from an iOS IPA using standard reverse-engineering tools. Without Row Level Security (RLS), that extracted key would give the holder unlimited read access to every message in the database for every user, and the ability to insert messages directly into the table, bypassing the FastAPI authentication layer entirely.

RLS is the Supabase/PostgreSQL mechanism that enforces data access at the database level, regardless of how a client connects. Even if someone has the anon key, they can only see rows that the policies explicitly permit.

---

## The architecture: two auth systems

Understanding why this setup required careful wiring comes down to understanding that the app has two separate authentication systems:

### 1. FastAPI JWT (primary auth)
- Used for all API calls (profiles, discovery, connections, etc.)
- A custom HS256 token generated by the backend on login/register
- Stored in Flutter's `flutter_secure_storage` on the device
- Sent as a `Bearer` token in the `Authorization` header of every HTTP request
- Verified by FastAPI's `dependencies.py` → `get_current_user()`

### 2. Supabase (realtime only)
- Used exclusively for the live message subscription in `chat_screen.dart`
- Initialised with the anon key in `main.dart`
- Does NOT use Supabase Auth — users never sign in via Supabase's own auth system

The problem this created: Supabase's RLS policies use `auth.uid()` to identify who is making a request. But `auth.uid()` works by reading the JWT that the client passes to Supabase. Since the app was only passing the anon key (no user identity), `auth.uid()` returned `NULL` for every realtime connection, meaning any identity-based policy would silently block all messages.

---

## The fix: wiring FastAPI JWT into Supabase Realtime

Three changes were made to close this gap:

### Backend — `app/security.py`
The FastAPI JWT payload was updated to include the `role` claim that Supabase requires to recognise a user as authenticated:

```python
payload = {
    'sub': str(user_id),       # user UUID — becomes auth.uid() in Supabase
    'role': 'authenticated',   # required by Supabase RLS
    'type': 'access',
    'iat': now,
    'exp': now + timedelta(minutes=settings.access_token_expire_minutes),
}
```

Without `role: authenticated`, Supabase treats every connection as anonymous even if a JWT is present.

### Supabase Dashboard — JWT Secret
In **Project Settings → Auth → JWT Settings**, the JWT Secret was changed to match the `JWT_SECRET_KEY` value from the backend `.env` file.

This tells Supabase: "trust and verify tokens signed with this secret". Before this change, Supabase had no way to validate the FastAPI tokens because it did not know the signing key. After this change, Supabase decodes the FastAPI JWT, reads the `sub` claim as the user's UUID, and makes it available as `auth.uid()` inside RLS policies.

**Important:** the JWT secret value must be identical on both sides — the exact same string. Length does not need to match any particular format; HS256 accepts keys of any length. A 64-character key from the backend and a 32-character default from Supabase would not match because they are different values, not because of their lengths. Always copy the backend `JWT_SECRET_KEY` value directly.

### Flutter — `lib/features/auth/providers/auth_provider.dart`
After every login, register, and on app startup (when restoring a session from storage), the FastAPI JWT is now passed to the Supabase realtime client:

```dart
Supabase.instance.client.realtime.setAuth(token);
```

On logout, it is cleared:

```dart
Supabase.instance.client.realtime.setAuth(null);
```

`setAuth` is a first-class method on the Supabase Dart SDK's `RealtimeClient`. It updates the authorisation token for all active and future channel subscriptions. Without this call, the realtime client would connect as anonymous regardless of whether the user was logged in.

## The RLS policies

### Policy 1: `participants can read their messages` (SELECT)

```sql
CREATE POLICY "participants can read their messages"
ON messages FOR SELECT
USING (
  EXISTS (
    SELECT 1 FROM matches
    WHERE matches.id = messages.match_id
      AND (matches.user_a_id = auth.uid() OR matches.user_b_id = auth.uid())
  )
);
```

**What it does:** A user can only read a message if there is a row in the `matches` table where their user ID appears as either `user_a_id` or `user_b_id`, and that match's ID matches the message's `match_id`.

**In plain English:** You can only see messages from conversations you are actually part of. If you are not in the match, you get zero rows back — not an error, just no data.

**Why `auth.uid()`:** This function reads the `sub` claim from the JWT that the client passes to Supabase. After the wiring described above, this equals the logged-in user's UUID from the FastAPI token.

### Policy 2: `no direct inserts` (INSERT)

```sql
CREATE POLICY "no direct inserts"
ON messages FOR INSERT
WITH CHECK (false);
```

**What it does:** Blocks all INSERT operations made through the Supabase client (anon key or user JWT). `WITH CHECK (false)` means the condition is never true, so no direct insert can ever succeed.

**Why this is safe for the app:** All message sending goes through the FastAPI endpoint `POST /messages/threads/{match_id}`. FastAPI uses the `SUPABASE_SERVICE_ROLE_KEY` on the backend to write to the database. The service role key bypasses all RLS policies entirely — it is a superuser-level key that only exists on the server and is never exposed to clients. So legitimate message sending still works; only direct client-side inserts are blocked.

## How RLS was enabled

RLS was enabled and policies were created by running the following SQL in the **Supabase Dashboard → SQL Editor**:

```sql
-- Step 1: Enable RLS on the messages table
ALTER TABLE messages ENABLE ROW LEVEL SECURITY;

-- Step 2: Restrict reads to match participants only
CREATE POLICY "participants can read their messages"
ON messages FOR SELECT
USING (
  EXISTS (
    SELECT 1 FROM matches
    WHERE matches.id = messages.match_id
      AND (matches.user_a_id = auth.uid() OR matches.user_b_id = auth.uid())
  )
);

-- Step 3: Block all direct inserts via client keys
CREATE POLICY "no direct inserts"
ON messages FOR INSERT
WITH CHECK (false);
```

## The rogue policy problem — and what to watch for on re-migration

During setup, a third policy was discovered that had been **auto-created by Supabase** when the `messages` table was first set up through the Dashboard's Table Editor UI:

| Policy name | Type |
|---|---|
| `anon_can_read_messages` | SELECT |

This policy granted the `anon` role unrestricted read access to all messages with no conditions. In PostgreSQL, multiple permissive policies on the same command (SELECT) are combined with **OR** — so even with the correct `participants can read their messages` policy in place, this rogue policy was overriding it and allowing anyone with the anon key to read everything.

It was removed with:

```sql
DROP POLICY "anon_can_read_messages" ON messages;
```

### Will this happen again on re-migration?

**It depends on how the table is recreated:**

- **If the table is recreated via SQL (Alembic migration, raw SQL script, Supabase CLI `supabase db push`):** No. Raw SQL does not trigger Supabase's policy wizard. The table will be created with no policies, and you must run the three SQL statements above manually or include them in the migration script.

- **If the table is created through the Supabase Dashboard Table Editor:** Yes. The Dashboard offers to create RLS policies when you create a table, and one of the default templates creates an open `anon` read policy. Always decline this offer, or drop any auto-created policies immediately after.

### Recommended: include policies in migration

To make this repeatable and safe, add the RLS setup directly into a migration file. Create a new Alembic migration or an `init_rls.sql` file that runs after any schema rebuild:

```sql
-- Run after schema is created to enforce message security
ALTER TABLE messages ENABLE ROW LEVEL SECURITY;

-- Drop any Supabase-auto-created open policies if they exist
DROP POLICY IF EXISTS "anon_can_read_messages" ON messages;

-- Correct policies
CREATE POLICY IF NOT EXISTS "participants can read their messages"
ON messages FOR SELECT
USING (
  EXISTS (
    SELECT 1 FROM matches
    WHERE matches.id = messages.match_id
      AND (matches.user_a_id = auth.uid() OR matches.user_b_id = auth.uid())
  )
);

CREATE POLICY IF NOT EXISTS "no direct inserts"
ON messages FOR INSERT
WITH CHECK (false);
```

The `DROP POLICY IF EXISTS` and `CREATE POLICY IF NOT EXISTS` syntax means this script is safe to run multiple times without errors.

## Verification checklist

Run these after any migration or Supabase reset to confirm everything is correct:

```sql
-- 1. RLS is enabled (relrowsecurity must be true)
SELECT relname, relrowsecurity 
FROM pg_class WHERE relname = 'messages';

-- 2. Exactly two policies exist (no extras)
SELECT policyname, cmd 
FROM pg_policies WHERE tablename = 'messages';

-- 3. anon role cannot bypass RLS (rolbypassrls must be false)
SELECT rolname, rolbypassrls 
FROM pg_roles WHERE rolname = 'anon';
```

Expected results:

| Check | Expected value |
|---|---|
| `relrowsecurity` | `true` |
| Policies listed | `participants can read their messages` (SELECT) and `no direct inserts` (INSERT) — exactly these two, nothing else |
| `rolbypassrls` for anon | `false` |

If you see more than two policies on the messages table, identify any extras and assess whether they are granting broader access than intended. Drop them if they are.

## JWT secret key length

The JWT secret does not have a required length. HS256 (HMAC-SHA256) accepts signing keys of any length. What matters is that the exact same string value is used on both sides:

- Backend `JWT_SECRET_KEY` in `.env`
- Supabase **Project Settings → Auth → JWT Settings → JWT Secret**

If the backend uses a 64-character key and you paste a different 32-character Supabase-generated value into the backend, they will not match and authentication will break — not because of length, but because the values are different. Always copy the value from one source and paste it into the other. The direction (backend → Supabase or Supabase → backend) does not matter as long as they end up identical.

For security, a minimum of 32 characters (256 bits) is recommended for HS256. Longer keys (64+ characters) are fine and provide a larger margin against brute-force attacks.
